# (블로그 초안) “요청 버튼 한 번”으로 앱 프로토타입이 만들어지는 날

어제는… 진짜 말 그대로 **“자동화가 자동화를 만든 날”**이었다.

우리가 만들고 있는 건 ‘10고개 앱빌더’다. 이름은 귀엽지만 목표는 꽤 야심차다.

“관리자가 앱 아이디어를 요청으로 올리면 → 웬디가 그걸 읽고 → 프로토타입을 만들어서 → 링크로 딱 던져주는”

이 흐름이 꿈이 아니라, 실제로 돌아가게 만드는 게 어제의 미션.

## 1) 먼저, 개발자는 믿음을 먹고 산다 (그래서 버그부터)
자동화는 멋있어 보이지만, 기본이 안 되어 있으면 그냥 ‘폭탄 돌리기’가 된다.
그래서 어제는 RLS 권한 이슈부터 잡았다.
- 버그리포트 쪽 권한 체크
- 프로젝트 저장이 제대로 안 되는 문제
- PC 프리뷰 UX(기본은 모바일, 버튼으로 PC 전환)

이런 것들을 하나씩 정리하면서 “아, 이제 서비스처럼 보이기 시작한다”가 나왔다.

## 2) 기능이 붙기 시작하면, 서비스는 갑자기 살아난다
어제 새로 붙인 기능들이 꽤 ‘서비스 핵심’에 가까웠다.
- 컬러 피커: 브랜드 컬러를 마음대로
- 테스터 기능: 지정된 사람은 무료로
- 결제 후 9단계 온보딩: 결제 후 이탈을 줄이는 긴장감 있는 UX
- 관리자 요청 페이지: 목록/상세/상태변경까지 운영 플로우
- “웬디에게 요청” 버튼
- 그리고 핵심: `/api/wendy/pending-requests`

이 API가 생기면서 ‘대기열’ 개념이 잡혔다.

## 3) 자동화는 보안이 없으면 망한다
“AI가 만들어주는 서비스”는 늘 위험이 따라온다.
그래서 보안 규칙도 같이 만들었다.
- 금지 키워드 필터링
- 의심스러운 요청이면 중단
- API 응답에 보안 지침 포함

개발 속도만 올리는 게 아니라, “운영 가능한 속도”로 올리는 느낌.

## 4) 그리고, 첫 프로토타입이 진짜로 튀어나왔다
가장 짜릿했던 순간.
존이 테스트로 프로젝트 요청을 올렸고, 웬디 조수가 프로토타입을 만들어서 배포 링크를 뽑아냈다.

결과: https://prototype-promotion.vercel.app

화면도 무려 5개.
- 랜딩
- 로그인
- 프로모션 목록/상세
- 마이페이지

와디즈 느낌 + 브랜드 컬러 `#3b3758`.
“아… 이거 진짜 되는 거네?”

## 5) 마지막으로, 운영 자동화까지
웬디가 “작업 중/완료” 상태로 바뀌면 프로젝트 상태도 자동으로 따라가게 했다.
- working → in_progress
- completed → completed
- 결과 URL 자동 저장

이렇게 되면 사람은 ‘관리’만 하고, 실행은 시스템이 한다.

## 6) 다음은?
이제 남은 건 딱 두 가지다.
- 테스터를 실제로 모으고, 피드백 루프 돌리기
- 결제/온보딩 흐름을 실전으로 다듬기

어제는 확실히 하나를 증명했다.

“앱을 만드는 시스템”은 생각보다 빨리 현실이 될 수 있다.
